# Database Migration Playbook (SQLite to Azure PostgreSQL)\n\n## Context\nRentipid is migrating its primary datastore from a local/disk-based SQLite database on Vercel to Azure Database for PostgreSQL Flexible Server to support high concurrency, row-level locking, and high availability.\n\n## Objective\nEnsure a seamless transition of live production data with minimal to zero downtime.\n\n## Phase A: Preparation (Staging)\n1. Provision Azure Database for PostgreSQL Flexible Server in the selected region.\n2. Configure firewall rules allowing Azure Container Apps (Backend API) to access the DB.\n3. Create the empty database schema using `npx prisma migrate deploy`.\n\n## Phase B: Data Replication Strategy\nBecause SQLite does not natively support Logical Replication (CDC) like Postgres-to-Postgres migrations do, the data migration must be handled via a bulk export/import script (ETL).\n\n1. **Maintenance Window**: Announce a 2-hour 'Read-Only' or maintenance window to users.\n2. **Freeze Vercel App**: Trigger the 'Emergency Freeze' feature flag (ADR-10) to stop new bookings or payments.\n3. **ETL Execution**: Run a custom Node.js script that pulls from SQLite and batches inserts into PostgreSQL.\n4. **Reconciliation Check**: Run `SELECT count(*) FROM table` queries on both databases to verify row counts match exactly for Users, Listings, Bookings, and Ledgers.\n\n## Phase C: Cutover\n1. Update the Vercel Production Environment Variables (and Azure Container Apps config) to point `DATABASE_URL` to the new Azure PostgreSQL connection string.\n2. Re-deploy the Vercel Frontend and Azure Backend to pick up the new connection strings.\n3. Disable the 'Emergency Freeze' feature flag.\n4. Run a sanity check: Create a test booking and verify it commits successfully.\n\n## Rollback Plan\nIf the PostgreSQL database fails during cutover:\n1. Revert the `DATABASE_URL` environment variable back to the SQLite path.\n2. Re-deploy the Vercel application.\n3. Manually refund any PayMongo transactions that occurred during the brief window PostgreSQL was active.
