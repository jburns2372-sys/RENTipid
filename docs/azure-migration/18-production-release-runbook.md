# Production Release & Rollback Runbook\n\n## Objective\nDefines the exact steps required to deploy the Azure Backend and how to roll it back safely in case of catastrophic failure.\n\n## Deployment (CI/CD)\n1. Merging to the `main` branch automatically triggers `.github/workflows/azure-deploy.yml`.\n2. The workflow builds the Docker image from `/apps/api` and pushes it to Azure Container Registry (ACR).\n3. Azure Container Apps automatically pulls the new image and spins up a new revision.\n\n## Emergency Rollback Strategy\nIf the newly deployed revision contains a critical bug (e.g., PayMongo webhooks begin failing):\n\n### Option 1: Azure Portal (Fastest)\n1. Log into the Azure Portal.\n2. Navigate to the `rentipid-api-app` Container App.\n3. Click **Revisions** in the left menu.\n4. Locate the previous (stable) revision, which will be marked `Inactive` or `0% traffic`.\n5. Change the traffic weighting of the stable revision back to `100%`.\n6. Change the traffic weighting of the broken revision to `0%`.\n\n### Option 2: Azure CLI\n```bash\n# List available revisions\naz containerapp revision list -n rentipid-api-app -g rg-rentipid-prod -o table\n\n# Direct 100% of traffic back to the old known-good revision\naz containerapp ingress traffic set -n rentipid-api-app -g rg-rentipid-prod --revision-weight <OLD_REVISION_NAME>=100\n```\n\n## Database Rollback Warning\nIf the broken deployment included a destructive Prisma Migration (`prisma migrate deploy`), rolling back the Container App revision **WILL NOT** roll back the database schema.\nTo roll back a database schema:\n1. Connect to the Azure PostgreSQL instance.\n2. Restore from the automated Azure Point-in-Time Restore (PITR) backup taken right before the deployment.
